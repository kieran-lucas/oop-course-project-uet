# UML Source-Level Coverage Audit

This document explains how the README class diagrams and supporting documentation map to the current `src/main/java` source tree. It separates real source-level design classes from generated bytecode artifacts so the diagrams stay readable and accurate.

---

## 1. Audit Basis

The audit is based on:

- `src/main/java/com/auction/**`
- `src/main/resources/db/migration/V1__initial_schema.sql` through `V17__wallet_transactions.sql`
- `build.gradle.kts`
- `README.md`
- `docs/SETUP.md`
- `docs/SCHEMA.md`
- `docs/BUSINESS_RULES.md`

The goal is design-level correctness, not a byte-for-byte AST dump.

---

## 2. Included Scope

Included in design-level UML/documentation coverage:

- Top-level Java source files in `src/main/java/com/auction`.
- Real nested source-level helpers, records, enums, and interfaces when they matter architecturally.
- Controllers, services, DAOs, DTOs, models, exceptions, utilities, JavaFX controllers, and pattern implementations.
- Runtime infrastructure that materially affects architecture: `DatabaseConfig`, Embedded PostgreSQL, PostgreSQL JDBC, HikariCP, Flyway, and JDBI.
- Flyway-managed database schema and migration history.

---

## 3. Excluded Scope

Excluded intentionally:

- Test classes under `src/test/java`.
- Compiled output under `build/classes`.
- Anonymous compiler-generated callback/listener/lambda classes.
- `package-info.java`.
- Runtime/generated folders such as `.gradle`, `build`, `data/postgres`, and `logs`.
- Repetitive getters/setters when they do not add design information.

These exclusions keep the class diagrams readable without hiding major architecture or business logic.

## 6. Runtime Architecture Coverage

The runtime architecture is:

```text
JavaFX Client
  -> RestClient / WebSocketClient
  -> Javalin HTTP routes / WebSocket routes
  -> JwtMiddleware
  -> Controllers
  -> Services
  -> DAOs / JDBI transactions
  -> HikariCP connection pool
  -> PostgreSQL JDBC driver
  -> Embedded PostgreSQL or external PostgreSQL
  -> Flyway-managed schema
```

Database infrastructure must not be reduced to only PostgreSQL. The source uses `DatabaseConfig` to initialize embedded or external PostgreSQL, configure HikariCP, run Flyway, and expose JDBI.

## 7. Design Pattern Coverage

| Pattern | Source files | Purpose |
|---|---|---|
| Factory | `UserFactory`, `ItemFactory`, `AuctionStateFactory` | Centralized construction for roles, item categories, and auction states. |
| State | `AuctionState`, concrete state classes, `AuctionStates` | Encapsulates behavior by auction lifecycle state. |
| Observer | `AuctionEventListener`, `AuctionEventManager`, `WebSocketObserver`, `AuctionWebSocketHandler` | Pushes bid/time/end events to connected clients. |
| Strategy | `AutoBidStrategy` | Encapsulates auto-bid chain execution and fairness rules. |
| DAO | `dao/*Dao` | Encapsulates database access through JDBI and SQL mapping. |

## 8. Database and Migration Coverage

Schema documentation must reflect the final Flyway state after migrations `V1` through `V17`:

| Area | Required documented detail |
|---|---|
| Users | role, balance, reserved balance, token version |
| Items | seller ownership, category fields, item status |
| Auctions | seller id, lifecycle statuses, current price, leading bidder, timestamps |
| Bids | manual/auto bid history |
| Auto-bid | increment, status, failure reason, registered time, unique auction/bidder pair |
| Deposits | pending/approved/rejected admin review flow |
| Password resets | admin-reviewed reset flow and one-pending-request rule |
| Notifications | persistent user notification feed |
| Wallet ledger | deposit, freeze, release, winner consume, seller payout, cancel release |

## 10. Verification Notes

The documentation is considered aligned when:

- README sections 1-6 mention HikariCP, Flyway, JDBI, PostgreSQL JDBC, reserved balance, wallet ledger, and executable fat JARs where relevant.
- `SCHEMA.md` describes the post-migration schema, not only the initial `V1` schema.
- `BUSINESS_RULES.md` describes behavior implemented by the current services and avoids claiming unsupported behavior such as item status changes during settlement unless verified in source.
- `SETUP.md` gives copy-paste-safe commands, especially for `JWT_SECRET`.
- README class diagrams preserve coverage of major packages without expanding every anonymous/generated class.
